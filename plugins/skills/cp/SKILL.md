# cp — commit & push

> 运行环境入口约定：本仓库的 `.claude/skills` 以 Claude Code 为源，示例默认写 `/tunan:*`。若同一 skill 在 Codex 中运行，所有面向 sponsor 的可复制入口在输出前改写为 `$tunan:*`；Claude Code 中保持 `/tunan:*`。

> **何时触发**：用户说 "cp" / "commit push" / "提交并推送" / "/tunan:cp"。

## 调用语法

```
/tunan:cp [开关] [-- <commit message>]
```

**开关**：

- `--no-push` — 只 commit 不 push（用户只说"提交"时；纯 commit 也可以走这条）
- `--files=<glob,...>` — 限定要 add 的文件（默认按当前 staged + unstaged 改动智能挑，但永远不用 `-A` / `.`）
- `--message=<text>` 或 `-- <text>` — 显式给 commit message，跳过自动起草
- `--allow-main`（别名 `--am`） — 允许直接推到 `main` / `master` / `dev` / `test`（默认会先警告确认）。`--am` 与 `--allow-main` 完全等价

**示例**：

```
/tunan:cp                                             # 标准：自动起 message → commit + push
/tunan:cp --no-push                                   # 只 commit
/tunan:cp -- 修复登录 token 过期未刷新                  # 显式 message
/tunan:cp --files=src/auth/**                         # 只 commit auth 目录
```

> 不接受 `--amend` / `--no-verify` / `--force` 等危险开关；遵循全局 CLAUDE.md 的 git-commit-push 纪律。

## 前置

- 假设代码已改完且用户已确认
- 不做额外的 lint / test，那是 `/tunan:test` 的事

## 流程

1. 并行执行：
   - `git status`（不要 `-uall`）
   - `git diff`（staged + unstaged）
   - `git log --oneline -5`（学习 commit message 风格）
2. 分析 diff，起草 commit message：
   - 1-2 句，聚焦 **why** 而不是 **what**
   - 跟随近期 commit 的语言（中文 / 英文）和格式
   - **禁止在 commit message 开头加 `@` 符号**。`@` 是 GitHub / Git 平台的 mention 语法，不是 commit 格式前缀。即使 `git log` 里已有的历史 commit 带 `@ ` 前缀（如 `@ fix(...):`），那些是错误示范，**绝对不要模仿**。正确的 conventional commit 格式直接以 type 开头：`fix(scope): 描述`、`feat(scope): 描述`
   - 不要带 emoji，不要写 "Generated by Claude" 之类的尾巴
3. 跳过明显敏感文件（`.env` / `credentials*` / 大二进制）；遇到则警告用户而不是直接加进来
4. 并行执行：
   - `git add <具体文件>`（不用 `git add -A` / `git add .`）
   - `git commit -m "..."`（HEREDOC 传消息）
5. push 前**硬检查**当前分支，取 `BRANCH=$(git rev-parse --abbrev-ref HEAD)`（下文 `--allow-main` 均含其别名 `--am`）：
   - 若 `BRANCH` 匹配 `dev` / `main` / `master` 且**无** `--allow-main`（或 `--am`） → **不要自动 push，也绝不在 chat 里手打选项菜单**。用平台的阻塞式提问工具对齐：Claude Code 用 `AskUserQuestion`（若其 schema 未加载，先 `ToolSearch` `select:AskUserQuestion`），Codex 用 `request_user_input`，Gemini / Pi 用 `ask_user`；仅当无阻塞工具或调用报错时才回退为 chat 编号列表。问 `当前分支 <BRANCH> 是 protected base，这次 push 怎么处理？`，给以下选项（默认预选最安全的第一项，标 `(Recommended)`）：
     - **改到新 feature 分支再推**（Recommended）：起一个分支名（据本次改动推断，或问用户）→ `git checkout -b <new-branch>` 带走本次 commit → `git push -u origin <new-branch>`，不污染 protected base。
     - **直接推到 `<BRANCH>`**：用户选此项即视为显式授权（等价补 `--am`）→ `git push -u origin <BRANCH>`。
     - **只保留 commit，先不 push**：停在已 commit 状态，结束。
   - 否则（feature 分支，或 protected base + `--allow-main` / `--am`）→ `git push -u origin <BRANCH>`（`-u` 确保追踪关系）
   - push 若被远端拒绝（non-fast-forward，远端已领先）→ 进入 **§自动同步重试**，自动 merge pull 后重推，不停下等用户
6. `git status` 验证 working tree clean、ahead 归零

## 自动同步重试（push 被 non-fast-forward 拒绝时）

push 因远端领先被拒时，**自动**走以下闭环（**merge pull**，产生 merge commit），不停下等用户（用户触发 cp 即已授权完成这次 push）：

1. `git fetch origin <BRANCH>`
2. **冲突预检**：`git diff --stat HEAD...origin/<BRANCH> -- <本次提交涉及的文件>`。与远端新增**无文件交集** → 大概率零冲突，直接进 merge。
3. `git pull --no-rebase origin <BRANCH>`（merge 远端到本地，保留本地与远端两边历史，产生 merge commit）
4. 按 merge 结果分流：
   - **无冲突**（merge 干净）→ `git push origin <BRANCH>`，完成。
   - **有冲突** → 逐个冲突文件分析：
     - 能确定**安全合并**的（双方改不同区域被 git 误判、或一方是 CI / 格式 / 版本号 / 依赖锁这类机械改动）→ 合并保留双方意图 → `git add <file>` → `git commit --no-edit`（收束 merge commit）→ push。
     - 涉及**同一逻辑的实质冲突**、无法确信正确合并 → `git merge --abort`，**停下报告**冲突文件 + 双方差异摘要，让用户裁决。**绝不瞎解冲突**（盲目自动解会引 bug）。
5. push 仍被拒（merge 期间远端又更新）→ 重复 1–4，**最多 3 轮**；3 轮仍失败 → 停下报告。

红线：全程**不 force push**、不 `--amend`、不 `--no-verify`。

## 失败处理

- pre-commit hook 失败 → 修问题 → **新建 commit**（绝不 `--amend`，不 `--no-verify`）
- push 被 reject（非 fast-forward）→ 走 **§自动同步重试**：自动 `git pull --no-rebase`（merge）→ 安全解决冲突 → 重新 push；仅当实质冲突无法安全自动解决时才停下报告。**不要** force push
- 当前在 `dev` / `main` / `master` 且无 `--allow-main`（或 `--am`） → 不自动推；走 §5 的**阻塞式提问工具**让用户选（改 feature 分支 / 直接推 / 只 commit）。**绝不**用 chat 手打选项菜单替代阻塞工具——那是仅在无阻塞工具或调用报错时的回退

## 不要做

- 不要 `git add -A` / `.`
- 不要 `--amend` / `--no-verify` / `--no-gpg-sign`
- 不要 force push
- 不要顺手做"清理"提交（删 unused、改 format）—— 只 commit 当前改动

## 输出

> ✅ 已 commit 并 push 到 `<branch>`。commit: `<short-sha> <subject>`
> （若触发自动同步重试：补一句 `已自动 merge 远端最新（merge commit 收束，冲突 X 处已安全合并）`）
