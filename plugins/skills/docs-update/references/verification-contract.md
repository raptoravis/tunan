# 核验输出契约

核验子代理对每条"可核验事实"对照活代码检查，返回结构化结果。**核验是只读的**——核验子代理只读代码、不改文档（修复由 Phase 4 的撰写子代理做）。

## 单条核验结果

```
{
  "claim": "<文档里的原始断言，如 `parseConfig(path: string): Config`>",
  "doc_file": "<出现该断言的文档路径>",
  "status": "ok | wrong | unverifiable",
  "evidence": "<在代码里找到的实际情况：真实签名/路径/端点，或'未找到对应符号'>"
}
```

- **ok** — 断言与代码一致。无需动作。
- **wrong** — 代码里存在对应物但与文档不符（签名变了、路径移了、端点改了）。`evidence` 必须给出**实际值**，供 Phase 4 修正时直接采用。
- **unverifiable** — 代码里找不到对应符号/路径/端点，无法判定真伪。可能是文档写错，也可能是动态/外部行为。保留 `<!-- VERIFY -->` 标记，不擅自删改。

## 核验手段优先级

1. 原生 Glob/Grep/Read——查符号是否存在、查文件路径是否存在、查路由/配置定义的字面文本。
2. 不要用 shell `grep`/`find`/`cat` 做这些——用原生工具，避免子代理工作流里的权限弹窗。

## 严禁

- 核验子代理不修改文档文件（那是撰写子代理 Phase 4 的职责）。
- 不把"我觉得应该对"当成 ok——只有在代码里实证一致才标 ok，找不到就 unverifiable。
- 不删 `unverifiable` 项对应的文档内容——标记保留，交人工。
