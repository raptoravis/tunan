# 标准文档类型目录与内容契约

`docs-update` 适配仓库**已有的**文档布局，但当某个标准文档缺失且项目明显需要它时，可按下表生成。每种类型给出"该覆盖什么"的内容契约——撰写子代理据此组织内容，并保证每条事实可在代码里证实。

下表是候选清单，不是必须全建。优先刷新已存在的文档；只在项目类型明显需要却缺失时才新建（例如一个发布到 npm 的库缺 usage 文档）。

## README.md

项目门面。覆盖：一句话定位、安装/获取方式、最小可用示例（必须是能跑通的真命令）、核心功能要点、指向更深文档的链接。核验重点：安装命令、示例代码、CLI 调用、徽章链接。

## ARCHITECTURE.md（或 docs/architecture.md）

系统结构。覆盖：顶层模块划分与职责、关键数据流/控制流、重要的目录布局、外部集成点。核验重点：模块路径、目录名、提到的入口文件确实存在。优先用 CodeGraph 取真实的调用/依赖边，而非凭印象描述。

## CONTRIBUTING.md

贡献流程。覆盖：本地起步（装依赖、跑测试、跑 lint 的真命令）、分支/提交约定、PR 流程、代码风格要点。核验重点：脚本命令（对照 `package.json` scripts / Makefile / 任务运行器）、目录约定。

## API / Reference 文档

库或服务的公共接口。覆盖：导出的函数/类/类型签名、参数与返回、HTTP 端点（方法+路径+请求/响应形状）、配置键。核验重点：**逐条**对照源码签名/路由定义；这是最容易过期、最需要核验的一类。

## Usage / Guide 文档

面向使用者的操作指南。覆盖：常见任务的步骤、配置示例、CLI 子命令。核验重点：命令、flag、配置键名与默认值。

## CHANGELOG.md

**不由本 skill 生成**。变更日志是发布流程产物（见 `release-notes` skill），docs-update 不触碰它。

## 撰写约束（所有类型通用）

- 只写能在代码里证实的事实。不确定的，写 `<!-- VERIFY: <claim> -->` 标记交给核验阶段，不要编造。
- 保留已有文档里仍准确的手写内容（除非 `--force`）——只补缺失、改过期。
- 代码块里的命令/路径必须是真实可用的，不写占位符式的假命令。
- 返回时附上本次新增/改动的**可核验事实清单**（path / signature / command / endpoint），供核验子代理逐条 fact-check。
